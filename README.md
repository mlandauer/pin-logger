# Pin Logger

[<img alt="crates.io" src="https://img.shields.io/crates/v/pin-logger.svg?style=for-the-badge&color=fc8d62&logo=rust" height="20">](https://crates.io/crates/pin-logger)
[<img alt="docs.rs" src="https://img.shields.io/badge/docs.rs-pin%20logger-66c2a5?style=for-the-badge&labelColor=555555&logo=docs.rs" height="20">](https://docs.rs/pin-logger)

A rust no-std no-alloc embedded-hal compatible library for embedded logging, but with digital output pins to mark bits of code so you can see what's happening while your device is running but without needing a normal console for log output.

## Why would anyone need this?

Let's say your power profiling the device you're working on. It's got lots going on and you want to understand what sections of code you should focus your optimising efforts on.

Something like the [Power Profiler Kit 2 (PPK2)](https://www.nordicsemi.com/Products/Development-hardware/Power-Profiler-Kit-2) is a great inexpensive tool for that. As well as measuring the power usage of your device it has 8 digital inputs that are captured at the same time. It's a logic analyser of sorts but what it really allows you to do is set output pins on your device in particular bits of code to mark certain critical sections that you want to see where they pop up on the power usage.

You can do this pretty easily but as soon as you have more than a few different sections to mark it can all become a bit cumbersome and error prone. You have to remember which pin corresponds to what and you probably changed it 5 minutes ago while you were working. It's a recipe for confusion and mistakes.

This library aims to make this process a little less error prone.

In your code you mark interesting bits with `pin_log!` messages:
```rust ignore
pin_log!("Entering main loop");
```

Automatically each message gets assigned a set of output pin states at build. You can see these in a auto-generated file:
```text
This file is automatically generated on each build
It shows the output pin states that corresponds to each pin_log! in your code

0 1 2 3                                
pins     name                          location
---------------------------------------------------------------------
1 0 0 0  Turning on display            crates/m5paper/src/main.rs:123
0 1 0 0  Sending display to sleep      crates/m5paper/src/main.rs:128
1 1 0 0  Start wifi                    crates/m5paper/src/main.rs:131
0 0 1 0  starting embassy_net          crates/m5paper/src/main.rs:146
1 0 1 0  Waiting for stack to come up  crates/m5paper/src/main.rs:160
0 1 1 0  Network connected             crates/m5paper/src/main.rs:162
1 1 1 0  Battery setup                 crates/m5paper/src/main.rs:176
0 0 0 1  Entering main loop            crates/m5paper/src/main.rs:184
```

It will also show these in the console output as the device is running.

Then, when you use your power profiler you can see when the pins change and see which section is worthy of attention. To figure out what it corresponds you read of the pin values and you can read off the corresponding message and which line of code it came from.

## Setup

In `Cargo.toml`

```toml
[dependencies]
pin_logger = "0.2"

[build-dependencies]
pin_logger = { version = "0.2", features = ["build"] }
```

In `build.rs` (create in the root of your project if you don't already have one)

```rust ignore
fn main() {
    // On build put a file in the root directory that shows which output pins correspond to which log message
    pin_logger::build::scan_source_for_pin_logs(std::path::Path::new("pin_logs.txt"));
}
```

Create the static for holding the logger so it's easily accessible everywhere
```rust ignore
pin_logger::global_static!(Output);
```

After you've initialised your embedded hardware but before you do any logging you setup the pins that will be used. On ESP this might look like:
```rust ignore
pin_logger::init!(
    Output::new(p.GPIO25, Level::Low, Default::default()),
    Output::new(p.GPIO32, Level::Low, Default::default()),
);
```
